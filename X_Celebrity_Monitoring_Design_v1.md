## 1. プロジェクト概要

### 1.1 背景と目的

出版社クライアントから、所属する著名人（タレント、スポーツ選手、文化人等）に関するX（旧Twitter）上の投稿・言及を定期的にモニタリングし、レポートとして受け取りたいという要望があった。

本プロジェクトでは、既存のX-AutoSummaryシステムで確立した技術基盤を活用し、概念実証（PoC）としてプロトタイプを構築する。PoCの結果をもとに、正式なサービス化の判断材料とする。

### 1.2 スコープ（PoC範囲）

| 対象（スコープ内） | 対象外（将来検討） |
|---|---|
| 特定キーワードによるツイート検索・収集 | リアルタイムアラート通知 |
| AIによる感情分析・カテゴリ分類 | 画像・動画コンテンツの分析 |
| 日次HTMLレポート生成 | ダッシュボードWebアプリ |
| メール配信（既存SMTP活用） | 複数クライアント同時運用 |
| 注目ツイートのピックアップ | 公式X API（有料）連携 |
| 週次サマリーレポート | 他SNS（Instagram等）対応 |

## 2. システム構成

### 2.1 全体アーキテクチャ

本システムは以下の4つのモジュールで構成される。各モジュールはX-AutoSummaryの既存実装をベースに改修する。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  ① 収集      │ →  │  ② 分析      │ →  │ ③ レポート   │ →  │  ④ 配信      │
│              │    │              │    │    生成       │    │              │
│ twikit       │    │ Gemini AI    │    │ HTML生成     │    │ Gmail SMTP   │
│ キーワード検索│    │ 感情分析     │    │ グラフ描画   │    │ スケジュール │
│ Cookie認証   │    │ カテゴリ分類 │    │ 統計集計     │    │ 実行         │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

### 2.2 X-AutoSummaryからの主な変更点

| 項目 | 既存（X-Summary） | 新規（モニタリング） |
|------|---|---|
| データ取得 | リスト内ツイート取得 | キーワード検索（人名・@・ハッシュタグ） |
| 対象設定 | 固定リストID | WebUIから動的に追加・編集 |
| AI分析 | カテゴリ分類＋要約 | 感情分析＋カテゴリ＋要約＋注目度判定 |
| レポート | テキストメール | HTMLレポート（グラフ付き） |
| 頻度 | 日次 | 日次＋週次サマリー |
| 表記ゆれ | なし | 漢字・かな・ローマ字の複数クエリ対応 |

## 3. 機能詳細設計

### 3.1 対象人物設定（WebUI）

Flaskベースの管理画面を提供し、非技術者でも対象著名人の追加・編集・削除を行える。

| 設定項目 | 詳細 |
|---|---|
| 表示名 | レポートに表示される名前（例：「大谷翔平」） |
| 検索キーワード | 複数指定可。表記ゆれ対応（例：大谷翔平, Shohei Ohtani, おおたにしょうへい） |
| Xアカウント | @メンション検索用（例：@shaboreveal）任意項目 |
| 除外キーワード | ノイズ除去用（例：同姓同名の別人に関する語句） |
| カテゴリタグ | タレント / スポーツ / 文化人 / 経営者 など自由設定 |
| レポート配信先 | メールアドレス（複数指定可） |
| 有効/無効 | 一時的にモニタリングを停止可能 |

### 3.2 ツイート収集

twikitライブラリを使用し、Cookie認証でXの検索APIにアクセスする。各対象人物について設定された検索キーワードをOR結合してクエリを構築する。

- **取得件数**: 対象人物あたり最大100件/日（調整可能）
- **取得対象**: 過去24時間以内のツイート（日次）、過去7日間（週次）
- **重複排除**: ツイートIDベースで既取得分をスキップ
- **RT・引用**: リツイート・引用ツイートも取得対象に含む

### 3.3 AI分析（Google Gemini）

収集したツイートをGoogle Gemini APIに送信し、以下の分析を実施する。

| 分析項目 | 内容 | 出力例 |
|---|---|---|
| 感情分析 | 各ツイートのセンチメント判定 | Positive / Negative / Neutral + スコア（0〜100） |
| カテゴリ分類 | 話題の種類を自動分類 | 新刊情報 / イベント / スキャンダル / 日常 / ファン反応 |
| 要約生成 | 全体の傾向を自然言語で要約 | 「本日は新刊発表に関する反応が中心で、概ね好意的...」 |
| 注目ツイート | 影響力の高いツイートを抽出 | いいね数・RT数の上位5件をピックアップ |
| トレンド検出 | 急増している話題を検出 | 「炎上リスク：○○に関する否定的言及が前日比300%増」 |

### 3.4 レポート生成

分析結果をHTML形式のレポートとして生成する。視認性を重視し、グラフや色分けを活用する。

**日次レポート構成：**

1. **エグゼクティブサマリー** — 全対象人物の概況を一覧表示
2. **人物別詳細** — 感情分析チャート、カテゴリ別件数、注目ツイートTop5
3. **アラート** — 急激な変化や炎上リスクの警告表示
4. **統計データ** — 総ツイート数、センチメント比率、前日比較

**週次レポート構成：**

1. **週間トレンド** — 日別推移グラフ、話題の変遷
2. **人物別ハイライト** — 週間で最も注目されたトピック
3. **比較分析** — 前週比、センチメント変化の推移

## 5. データフロー

日次処理の流れ：

1. タスクスケジューラが毎朝 8:00 に Pythonスクリプトを起動
2. 設定ファイル（JSON）から対象人物リストを読み込み
3. 各人物のキーワードで twikit 検索を実行（間隔を空けてレート制限回避）
4. 取得ツイートを JSON に保存（重複チェック済み）
5. Gemini API に一括送信して分析（感情分析・分類・要約）
6. 分析結果をもとに HTML レポートを生成
7. Gmail SMTP でレポートを配信先にメール送信
8. ログファイルに実行結果を記録

## 6. リスクと対策

| リスク | 影響 | 対策（PoC） | 対策（正式版） |
|---|---|---|---|
| Cookie期限切れ | ツイート取得が停止 | 手動でCookie再取得（1〜2週間に1回程度） | 公式APIに移行 |
| レート制限 | 一時的に取得不可 | 検索間隔の調整、対象人物数を制限 | 公式APIの上限内運用 |
| X利用規約 | スクレイピング禁止、法的リスク | 社内PoCに限定、商用利用しない | 公式APIライセンス取得 |
| Gemini API制限 | 分析処理が停止 | 複数アカウント切替、モデルバージョン変更 | 有料プランに移行 |
| 精度問題 | 誤分類・誤判定 | 人手によるレビュー、プロンプト改善 | ファインチューニング検討 |
| 同姓同名 | 無関係ツイート混入 | 除外キーワード設定 | 高度なフィルタリングAI |

## 8. 正式版への移行判断基準

PoCの評価後、以下の基準をもとに正式サービス化を判断する。

- クライアントがレポート内容・頻度に満足しているか
- 分析精度（感情分析・カテゴリ分類）が実用レベルか
- 運用負荷（Cookie更新頻度等）が許容範囲か
- 正式版の月額コスト（$220〜5,050）に対するクライアントの予算感

**正式版で追加検討する機能：**

- 公式X API（Basic/Pro）への移行による安定運用
- リアルタイム通知（炎上検知時の即座アラート）
- Webダッシュボード（過去レポートの閲覧・検索）
- 複数クライアント対応（マルチテナント化）
- 他SNS（Instagram / Facebook等）対応
- PDFレポート自動生成（ブランディング対応）
